Interacción práctica con contratos en Aave V3.

Leer, aprobar, depositar, pedir prestado, repagar y retirar.

Integración

Contratos mínimos que necesitamos

Para una integración básica no necesitamos entender todo el protocolo, solo lo esencial.

Contratos clave

  • PoolAddressesProvider — registro para descubrir direcciones del mercado.
  • Pool — contrato principal para operaciones del usuario.
  • ERC-20 del activo subyacente — para approve y balances.
  • Opcional: contratos o datos auxiliares para leer estado del usuario y del mercado.

Nota importante: no hardcodear direcciones sin verificarlas en documentación oficial o en el registro del mercado.

Mental model

El flujo mental del desarrollador

Antes de escribir código, conviene pensar en este orden:

  • ¿Qué mercado voy a usar?
  • ¿Cómo obtengo el Pool correcto?
  • ¿Qué token voy a depositar?
  • ¿Debo aprobar gasto antes de interactuar?
  • ¿Mi depósito estará habilitado como colateral para pedir prestado?
«En DeFi, primero resuelves direcciones y permisos; luego ejecutas lógica.» Idea clave

Pool

Descubrir el Pool correctamente

La forma recomendada es obtener el Pool desde PoolAddressesProvider.

Puntos clave

  • PoolAddressesProvider es el registro de direcciones del protocolo para un mercado concreto.
  • El Pool es un proxy y su dirección operativa debe resolverse desde el provider del mercado.
  • Esto evita integraciones frágiles y reduce errores por direcciones obsoletas.

Referencia: aave.com/docs — direcciones por red y mercado.

Solidity

Interfaces mínimas para la práctica

Interfaces que sí vale la pena enseñar en una primera práctica:

  • IERC20 con approve, balanceOf y transferFrom (conceptualmente).
  • IPool con supply, borrow, repay, withdraw.
  • Opcional: setUserUseReserveAsCollateral si quieres explicar colateral explícito.

IPool

Interfaz mínima de Pool

IPool.sol Foundry 
interface IPool {
    function supply(
        address asset,
        uint256 amount,
        address onBehalfOf,
        uint16 referralCode
    ) external;

    function borrow(
        address asset,
        uint256 amount,
        uint256 interestRateMode,
        uint16 referralCode,
        address onBehalfOf
    ) external;

    function repay(
        address asset,
        uint256 amount,
        uint256 interestRateMode,
        address onBehalfOf
    ) external returns (uint256);

    function withdraw(
        address asset,
        uint256 amount,
        address to
    ) external returns (uint256);
}

Estas son las cuatro funciones mínimas que cubren casi toda la práctica inicial.

Permisos

Primer paso: approve

Antes de hacer supply o repay, el token ERC-20 debe dar permiso al Pool para mover fondos.

Puntos clave

  • approve(pool, amount) se hace en el token, no en Aave.
  • Sin allowance suficiente, supply y repay fallan.
  • Esto enseña una idea central de DeFi: permisos explícitos entre contratos.

Supply

Supply: primera interacción real

Flujo práctico:

  • El usuario ya tiene un ERC-20 compatible.
  • Hace approve al Pool.
  • Llama a pool.supply(asset, amount, onBehalfOf, referralCode).
  • Recibe aTokens que representan su depósito y acumulan interés con el tiempo.
«Supply es el punto de entrada más simple para entender Aave.» Idea clave

Diagrama · Liquidez

Cómo fluye el capital

Borrow

Borrow: segunda interacción real

Después de depositar y usar ese activo como colateral, el usuario puede pedir prestado otro activo.

Puntos clave

  • borrow exige una posición suficientemente colateralizada.
  • Al pedir prestado, el usuario recibe el activo y se crean debt tokens que representan la deuda.
  • interestRateMode define el tipo de deuda soportado por el mercado.

Repay

Repay: cerrar o reducir deuda

Repagar reduce o elimina la deuda del usuario.

Flujo

  • Hacer approve del activo prestado al Pool.
  • Llamar a repay con asset, amount, rateMode y onBehalfOf.
  • El protocolo quema los debt tokens equivalentes a la cantidad repagada.

Idea clave: repay también es un flujo de ERC-20 + allowance + llamada al Pool.

Withdraw

Withdraw: recuperar el depósito

El usuario puede retirar su depósito solo si hay liquidez disponible y si su posición sigue siendo segura tras el retiro.

Puntos clave

  • withdraw redime aTokens por el activo subyacente.
  • No todo depósito es inmediatamente retirable si todavía respalda una deuda activa.
  • Este punto ayuda a entender por qué colateral y deuda están conectados.

Wrapper

Wrapper mínimo para la práctica

Propón un contrato educativo, no productivo, con cuatro métodos:

  • deposit(asset, amount)
  • borrow(asset, amount, rateMode)
  • repay(asset, amount, rateMode)
  • withdraw(asset, amount)

Objetivo de esta versión: encapsular el flujo básico sin entrar todavía en optimizaciones, permisos avanzados o manejo complejo de errores.

Demo

Estructura del contrato demo

AaveBasicInteractor.sol Foundry 
contract AaveBasicInteractor {
    IPool public pool;

    constructor(address poolAddress) {
        pool = IPool(poolAddress);
    }
}

Luego agregarás funciones que reciban tokens, hagan approve, llamen a supply o repay, y permitan borrow o withdraw.

deposit

Función demo: deposit

AaveBasicInteractor.sol · deposit Foundry 
function deposit(address asset, uint256 amount) external {
    IERC20(asset).transferFrom(msg.sender, address(this), amount);
    IERC20(asset).approve(address(pool), amount);
    pool.supply(asset, amount, msg.sender, 0);
}

Puntos clave

  • El contrato recibe tokens del usuario con transferFrom.
  • Luego da permiso al Pool.
  • Finalmente hace supply en nombre del usuario.

borrow

Función demo: borrow

AaveBasicInteractor.sol · borrow Foundry 
function borrow(address asset, uint256 amount, uint256 rateMode) external {
    pool.borrow(asset, amount, rateMode, 0, msg.sender);
}

Puntos clave

  • borrow no necesita approve porque el usuario está recibiendo fondos, no entregándolos.
  • Pero sí exige colateral suficiente y configuración compatible del mercado.

repay

Función demo: repay

AaveBasicInteractor.sol · repay Foundry 
function repay(address asset, uint256 amount, uint256 rateMode) external {
    IERC20(asset).transferFrom(msg.sender, address(this), amount);
    IERC20(asset).approve(address(pool), amount);
    pool.repay(asset, amount, rateMode, msg.sender);
}

Puntos clave

  • El patrón es muy parecido a deposit.
  • La diferencia es que aquí no se mintean aTokens; se reduce deuda y se queman debt tokens equivalentes.

withdraw

Función demo: withdraw

AaveBasicInteractor.sol · withdraw Foundry 
function withdraw(address asset, uint256 amount) external {
    pool.withdraw(asset, amount, msg.sender);
}

Punto clave: withdraw intenta redimir el depósito del usuario, pero solo funcionará si la posición resultante sigue siendo segura y existe liquidez disponible.

GHO

GHO en la práctica

GHO se entiende como un caso especial de borrow.

Puntos clave

  • En el Aave V3 Ethereum Market, mintear GHO ocurre mediante la función borrow.
  • Primero se suministra colateral elegible.
  • Luego se pide prestado GHO igual que cualquier otra reserva, aunque GHO usa el modelo de facilitator del ecosistema.
«Mintear GHO, desde la perspectiva del integrador, se parece mucho a pedir prestado un activo en Aave.» Idea clave

Taller

Ejercicio práctico guiado

Caso simple de práctica:

  1. Depositar WETH o un token de prueba compatible.
  2. Verificar que el usuario recibe aTokens.
  3. Pedir prestado un activo pequeño o GHO, si el mercado y el colateral lo permiten.
  4. Repagar la deuda.
  5. Retirar el colateral restante.

Errores

Errores comunes en la implementación

  • Usar una dirección de Pool incorrecta en lugar de resolverla desde PoolAddressesProvider.
  • Olvidar approve antes de supply o repay.
  • Intentar withdraw mientras la posición sigue sosteniendo una deuda insegura.
  • Pedir prestado sin entender colateral, rate mode o límites del mercado.

Resumen

Qué debe recordar hoy

  • Pool es el contrato principal para casi toda interacción del usuario.
  • El flujo básico es: approve → supply → borrow → repay → withdraw.
  • PoolAddressesProvider ayuda a encontrar la dirección correcta del Pool.
  • GHO se puede enseñar como un caso práctico de borrow sobre Aave V3 Ethereum Market.
